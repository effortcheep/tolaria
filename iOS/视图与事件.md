# 视图与事件

> **UIView 的 layoutSubviews、drawRect 什么时候会被调用？和 setFrame、setBounds 有什么关系？另外，当你点击屏幕上一个按钮，iOS 底层是怎么找到这个按钮并响应的？**

---

## 一、layoutSubviews 的调用时机

`layoutSubviews` 是 UIView 的布局回调，**不能直接调用**，由系统在以下场景触发：

1. **init 初始化时**（如果 frame 不为 zero）
2. **setFrame / setBounds 改变了 size**
3. **addSubview 添加子视图时**
4. **UIScrollView 滚动时**
5. **屏幕旋转导致 bounds 变化**
6. **直接调用 setNeedsLayout → 下一个 RunLoop 周期**
7. **调用 layoutIfNeeded 立即触发**

### setFrame / setBounds 与 layoutSubviews 的关系

```objc
// setFrame 核心逻辑（简化）
- (void)setFrame:(CGRect)frame {
    self.center = CGPointMake(CGRectGetMidX(frame), CGRectGetMidY(frame));
    self.bounds = CGRectMake(0, 0, frame.size.width, frame.size.height);
    // 如果 size 发生变化 → 标记需要 layout
    if (sizeChanged) {
        [self setNeedsLayout];
    }
}

// setBounds 核心逻辑（简化）
- (void)setBounds:(CGRect)bounds {
    // 改变内部坐标系原点（影响子视图定位）
    // 改变 size → 触发 setNeedsLayout
}
```

**关键区别：**
- `setFrame`：改变 view 在**父视图坐标系**中的位置和大小
- `setBounds`：改变 view **自身坐标系**的原点和大小
- 两者只要 **size 发生变化**，都会触发 `setNeedsLayout` → `layoutSubviews`

## 二、drawRect 的调用时机

`drawRect` 是 UIView 的绘制回调，用于**自定义2D绘图**：

1. **首次显示时**（视图可见且实现了 drawRect）
2. **调用 setNeedsDisplay / setNeedsDisplayRect 后**，下一个 RunLoop 周期
3. **bounds 的 size 发生变化**（如果 contentMode 不是 Redraw）

### drawRect 与 layoutSubviews 的区别

| | layoutSubviews | drawRect |
|---|---|---|
| **职责** | 布局子视图的 frame | 自定义内容绘制 |
| **触发方式** | setNeedsLayout | setNeedsDisplay |
| **执行时机** | layout pass | display pass |
| **调用顺序** | 先 | 后 |
| **性能影响** | 影响布局计算 | 影响 GPU 渲染 |

**注意：** 如果你没有重写 `drawRect`，UIView 不会创建底层的 Graphics Context，能节省内存。

## 三、Hit Testing —— iOS 怎么找到被点击的按钮

### 3.1 事件传递流程

```
Hardware Touch
    ↓
IOKit.framework → SpringBoard → App 进程
    ↓
UIApplication.sendEvent(_:)
    ↓
UIWindow.sendEvent(_:)
    ↓
hitTest:withEvent:  ← 核心：找到最精确的 view
    ↓
pointInside:withEvent:  ← 逐层判断触摸点是否在视图内
    ↓
目标 view 的 touchesBegan / addTarget action
```

### 3.2 hitTest:withEvent: 的完整算法

```objc
- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {
    // 1. 三个条件直接返回 nil，不处理该事件
    if (self.hidden || !self.userInteractionEnabled || self.alpha < 0.01) {
        return nil;
    }
    
    // 2. 判断触摸点是否在自己范围内
    if (![self pointInside:point withEvent:event]) {
        return nil;
    }
    
    // 3. 倒序遍历子视图（后添加的在上层，优先响应）
    for (UIView *subview in [self.subviews reverseObjectEnumerator]) {
        CGPoint convertedPoint = [subview convertPoint:point fromView:self];
        UIView *hitView = [subview hitTest:convertedPoint withEvent:event];
        if (hitView) {
            return hitView;  // 找到了，返回最深层的 view
        }
    }
    
    // 4. 没有子视图能处理，自己就是最终响应者
    return self;
}
```

### 3.3 图解 Hit Testing 过程

```
触摸点 (★)

    ┌──────────────────────── Window ────────────────────────┐
    │                                                        │
    │    ┌──────────── A (pointInside? ✓) ────────────┐     │
    │    │                                             │     │
    │    │    ┌─── B (✓) ───┐                          │     │
    │    │    │              │        ┌─── D (✗) ───┐  │     │
    │    │    │              │        │              │  │     │
    │    │    │   ★ C (✓)   │        │   (空,无触摸) │  │     │
    │    │    │   ← 命中!    │        │              │  │     │
    │    │    │              │        │   ✗ pointInside│  │     │
    │    │    └──────────────┘        │     返回 NO   │  │     │
    │    │                            │              │  │     │
    │    │                            └──────────────┘  │     │
    │    │                                              │     │
    │    └──────────────────────────────────────────────┘     │
    │                                                        │
    │  ★ 触摸点在左侧 B 区域内，不在右侧 D 区域内             │
    │                                                        │
    └────────────────────────────────────────────────────────┘

遍历顺序: Window → A → B → C(命中!) 
         Window → A → D(✗ pointInside返回NO, 直接跳过, 不递归)

结果: C 是 hit-test view
```

### 3.4 响应者链（Responder Chain）

找到 hit-test view 后，事件沿响应者链传递：

```
UIButton (hit-test view)
    ↓ touchesBegan 未处理
UIView (superview)
    ↓
UIViewController
    ↓
UIWindow
    ↓
UIApplication
    ↓
AppDelegate → 丢弃事件
```

如果某个 responder 实现了对应 touch 方法且不调用 `[super touchesBegan:]`，事件传递中断。

### 3.5 影响 Hit Testing 的属性

| 属性 | 作用 |
|---|---|
| `userInteractionEnabled = NO` | 不参与 hitTest，子视图也不响应 |
| `hidden = YES` | 不参与 hitTest |
| `alpha < 0.01` | 不参与 hitTest |
| `clipsToBounds` | 超出范围的子视图不响应 |
| `pointInside:withEvent:` 重写 | 可自定义不规则点击区域 |
| `hitTest:withEvent:` 重写 | 可改变事件目标（穿透点击等） |

### 3.6 实际应用场景

```objc
// 场景：扩大按钮点击区域（按钮太小不好点）
- (BOOL)pointInside:(CGPoint)point withEvent:(UIEvent *)event {
    CGFloat margin = 10.0;
    CGRect area = CGRectInset(self.bounds, -margin, -margin);
    return CGRectContainsPoint(area, point);
}

// 场景：点击穿透（让事件传递到下层 view）
- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {
    UIView *hitView = [super hitTest:point withEvent:event];
    if (hitView == self) {
        return nil;  // 不处理，穿透到下层
    }
    return hitView;
}
```

## 四、手势识别器（UIGestureRecognizer）的介入

### 4.1 手势识别器与 Hit Testing 的关系

手势识别器**不影响** hitTest 的查找过程，但会在事件到达响应者链**之前**或**同时**拦截触摸事件。

```
Touch Event
    ↓
hitTest:withEvent: (找到 hit-test view)
    ↓
手势识别器 持有该 view 或其 superview 上的 recognizer 开始识别
    ↓
┌─────────────────────────────────────────────┐
│ 手势识别成功 → touchesCancelled 发送给 view  │
│ 手势识别失败 → touchesBegan/Ended 正常传递   │
└─────────────────────────────────────────────┘
```

### 4.2 手势识别器的优先级

手势识别器分为**离散型（Discrete）** 和 **连续型（Continuous）** 两类：

| 类型 | 代表 | 行为 |
|---|---|---|
| 离散型 | UITapGestureRecognizer, UISwipeGestureRecognizer | 识别成功后一次性触发，状态变为 Recognized |
| 连续型 | UIPanGestureRecognizer, UILongPressGestureRecognizer | 持续跟踪，状态从 Began → Changed → Ended/Cancelled |

**关键：** 手势识别器的默认行为是**延迟发送触摸事件给 view**。在识别期间，触摸事件会被"暂存"，直到：

- 手势识别成功 → 系统调用 `[view touchesCancelled:withEvent:]`，view 不再收到后续触摸
- 手势识别失败 → 系统调用 `[view touchesBegan:withEvent:]`，触摸事件正常传递

### 4.3 手势识别器与 touchesBegan 的时序

```objc
// 假设 view 上同时有 tap 手势和 touchesBegan 处理

// 默认行为（delaysTouchesBegan = NO）：
touchesBegan:     ← 立即调用
  ... 识别中 ...
touchesCancelled: ← 如果手势识别成功，取消 view 的触摸处理
// 或
touchesEnded:     ← 如果手势识别失败，view 正常处理

// 设置 delaysTouchesBegan = YES：
  ... 识别中 ...   ← 不会调用 touchesBegan
// 手势成功 → 不调用 touchesBegan，直接触发手势 action
// 手势失败 → 调用 touchesBegan，正常走 touches 流程
```

### 4.4 requireGestureRecognizerToFail —— 手势依赖

可以设置一个手势必须等另一个手势**失败**后才能触发：

```objc
// 单击必须等双击失败后才触发
[tap1 requireGestureRecognizerToFail:doubleTap];

// 长按必须等点击失败后才触发
[longPress requireGestureRecognizerToFail:tap];
```

时序：
```
触摸开始
    ↓
doubleTap 识别中...
    ↓
第二次点击没来 → doubleTap 失败 (State = Failed)
    ↓
tap1 才开始识别 → 成功 (State = Recognized)
    ↓
tap1 action 触发
```

### 4.5 canPrevent / canBePreventedBy 机制

手势识别器之间有**优先级协商**：

```objc
// UIGestureRecognizer 的方法
- (BOOL)canPreventGestureRecognizer:(UIGestureRecognizer *)preventedGestureRecognizer;
- (BOOL)canBePreventedByGestureRecognizer:(UIGestureRecognizer *)preventingGestureRecognizer;
```

默认行为：
- **UIPanGestureRecognizer** 可以阻止 **UITapGestureRecognizer**
- **UISwipeGestureRecognizer** 可以阻止 **UIPanGestureRecognizer**

自定义示例：
```objc
// 自定义手势：阻止 pan 手势
- (BOOL)canPreventGestureRecognizer:(UIGestureRecognizer *)other {
    if ([other isKindOfClass:[UIPanGestureRecognizer class]]) {
        return YES;
    }
    return [super canPreventGestureRecognizer:other];
}
```

### 4.6 手势识别器作用于父视图的情况

手势识别器可以加在**父视图**上，但子视图仍可响应：

```objc
// 父视图上有 pan 手势
UIPanGestureRecognizer *pan = [[UIPanGestureRecognizer alloc] initWithTarget:self action:@selector(handlePan:)];
[parentView addGestureRecognizer:pan];

// 子视图有 button
UIButton *button = [UIButton buttonWithType:UIButtonTypeSystem];
[parentView addSubview:button];

// 结果：
// - 点击 button → button 的 action 触发（button 的 hit-test view 是自己）
// - 在 button 上滑动 → pan 手势识别成功 → button 收到 touchesCancelled
```

这是因为在 hitTest 阶段，button 是 hit-test view，但手势识别器挂在 parentView 上，两者可以同时工作。

### 4.7 总结：事件分发的完整顺序

```
Hardware Touch
    ↓
UIApplication → UIWindow
    ↓
hitTest:withEvent: (找到 hit-test view)
    ↓
hit-test view 上的所有手势识别器开始识别
（包括 superview 上的手势识别器）
    ↓
┌─────────────────────────────────────┐
│ 手势识别成功？                        │
│  YES → touchesCancelled, 执行 action │
│  NO  → touchesBegan/Ended 正常传递   │
└─────────────────────────────────────┘
    ↓
Responder Chain 传递
```

## 五、总结

```
setFrame/setBounds (size变化)
    → setNeedsLayout
    → layoutSubviews (布局子视图 frame)
    → setNeedsDisplay
    → drawRect (自定义绘制)

Touch Event
    → hitTest:withEvent: (找到最精确 view)
    → pointInside:withEvent: (逐层判断)
    → UIGestureRecognizer 识别 (成功则 touchesCancelled)
    → Responder Chain (传递处理)
```
