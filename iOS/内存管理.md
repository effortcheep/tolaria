# 内存管理

> **iOS 的内存管理机制是什么？什么情况下会产生循环引用？怎么解决？请举几个常见的循环引用场景。**

## 一、iOS 内存管理机制

### 1. ARC（Automatic Reference Counting）

iOS 使用 **引用计数（Reference Counting）** 作为内存管理的核心机制。自 iOS 5 起，编译器自动插入 `retain`/`release`/`autorelease` 调用，开发者不再需要手动管理引用计数，称为 **ARC**。

**核心规则：**
- 每个对象内部维护一个引用计数器
- 创建（`alloc`/`new`/`copy`/`mutableCopy`）时计数 = 1
- 有新的强引用指向时计数 +1
- 强引用移除时计数 -1
- 计数归零时对象立即被销毁，内存被回收

### 2. 内存区域

| 区域 | 存储内容 | 特点 |
|------|---------|------|
| **栈区（Stack）** | 局部变量、函数参数 | 自动分配回收，速度快，空间小 |
| **堆区（Heap）** | OC 对象（alloc/new） | 手动分配（ARC 自动管理），速度较慢 |
| **全局区/静态区** | 全局变量、静态变量 | 程序启动时分配，结束时回收 |
| **常量区** | 字符串常量等 | 只读 |
| **代码区** | 编译后的二进制代码 | 只读 |

### 3. MRC vs ARC 对比

| | MRC | ARC |
|---|-----|-----|
| 引用计数管理 | 手动 retain/release | 编译器自动插入 |
| 关键字 | N/A | `strong`、`weak`、`unsafe_unretained`、`copy` |
| 循环引用检测 | 手动排查 | 需开发者自行避免，编译器辅助警告 |

## 二、什么是循环引用（Retain Cycle）

当 **对象 A 强引用对象 B，同时对象 B 也强引用对象 A** 时，两者的引用计数永远不会降为 0，导致内存无法被回收，这就是**循环引用**。

更广泛地说，只要引用链形成一个**闭环**（A → B → C → A），所有涉及的对象都无法被释放。

---

## 三、如何解决循环引用

### 1. 使用 `weak`（弱引用）

`weak` 不会增加对象的引用计数，当指向的对象被销毁时，`weak` 指针会自动置为 `nil`，避免悬垂指针。

```objc
// OC
__weak typeof(self) weakSelf = self;
self.block = ^{
    __strong typeof(weakSelf) strongSelf = weakSelf; // 防止提前释放
    [strongSelf doSomething];
};
```

```swift
// Swift
class MyClass {
    var closure: (() -> Void)?
    
    func setup() {
        closure = { [weak self] in
            guard let self = self else { return }
            self.doSomething()
        }
    }
}
```

### 2. 使用 `unowned`（Swift）

适用于确定闭包捕获的对象生命周期**不会短于**闭包本身的情况。不增加引用计数，对象销毁后指针**不会置 nil**（访问已释放对象会崩溃）。

```swift
closure = { [unowned self] in
    self.doSomething()
}
```

### 3. 手动打破循环

在合适的时机将某个引用置为 `nil`，或者使用 `delegate` 模式时将 delegate 声明为 `weak`。

## 四、常见循环引用场景

### 场景 1：闭包（Block / Closure）捕获 self

**最经典的循环引用。**

```objc
// 错误 ❌
self.block = ^{
    [self doSomething]; // self → block → self
};
```

**解决：**
```objc
// 正确 ✅
__weak typeof(self) weakSelf = self;
self.block = ^{
    [weakSelf doSomething];
};
```

**原理：** self 持有 block 属性 → block 捕获了 self → 形成环。

---

### 场景 2：Delegate 使用 strong 而非 weak

```objc
// 错误 ❌
@property (nonatomic, strong) id<MyDelegate> delegate;
```

**解决：**
```objc
// 正确 ✅
@property (nonatomic, weak) id<MyDelegate> delegate;
```

**原理：** ViewController 持有 tableView → tableView 的 delegate 指向 ViewController → 环。

---

### 场景 3：NSTimer 强引用 target

```objc
// 错误 ❌
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                              target:self
                                            selector:@selector(tick)
                                            userInfo:nil
                                             repeats:YES];
```

**原理：** RunLoop 强引用 timer → timer 强引用 target（self）→ self 持有 timer → 三方循环。

**解决：**
- iOS 10+ 使用 block 版本 timer：
```objc
__weak typeof(self) weakSelf = self;
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0 repeats:YES block:^(NSTimer *timer) {
    [weakSelf tick];
}];
```
- 使用中间代理对象（NSProxy 子类）转发消息
- 在 `dealloc` 或合适时机调用 `[self.timer invalidate]`

---

### 场景 4：WKWebView 的 messageHandler

```objc
// 错误 ❌ — WKScriptMessageHandler 强持有 self
[self.webView.configuration.userContentController addScriptMessageHandler:self name:@"callback"];
```

**解决：**
```objc
// 正确 ✅ — 使用弱引用代理
[self.webView.configuration.userContentController
    addScriptMessageHandler:[[WeakScriptMessageDelegate alloc] initWithDelegate:self]
    name:@"callback"];
```

---

### 场景 5：相互持有的两个对象

```objc
// 对象 A
@property (nonatomic, strong) Person *partner;

// 错误 ❌
personA.partner = personB;
personB.partner = personA; // A → B → A
```

**解决：** 其中一方使用 `weak`：
```objc
@property (nonatomic, weak) Person *partner;
```

## 五、检测循环引用的方法

| 方法 | 说明 |
|------|------|
| **Xcode Memory Graph Debugger** | 运行时查看对象引用关系图，直观发现环 |
| **Instruments → Leaks** | 自动检测泄漏的内存块 |
| **Instruments → Allocations** | 观察对象是否在预期时机被释放 |
| **Debug Memory Graph** | 在 Xcode 底部 Debug Navigator 中查看内存图 |
| **代码检测** | 在 `dealloc` 中加断点/日志，验证对象是否被释放 |

---

## 六、要点总结

1. **ARC 不等于没有内存问题**，循环引用需要开发者自己避免
2. **核心解法：** `weak` 打破强引用环
3. **闭包是最常见的来源** — 养成 `[weak self]` / `__weak` 的习惯
4. **delegate 始终声明为 weak**
5. **NSTimer 是隐藏的陷阱** — 注意其对 target 的强引用
