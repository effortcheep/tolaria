# 跨域

## 什么是同源策略

**同源策略（Same-Origin Policy）** 是浏览器的安全机制，限制一个源（origin）的文档或脚本与另一个源的资源进行交互。

**同源三要素**：协议 + 域名 + 端口，三者完全相同才算同源。

```
http://example.com:80/path
 └── 协议   └── 域名  └── 端口
```

| URL A | URL B | 是否同源 | 原因 |
|---|---|---|---|
| `http://a.com` | `https://a.com` | ❌ | 协议不同 |
| `http://a.com` | `http://b.com` | ❌ | 域名不同 |
| `http://a.com:80` | `http://a.com:8080` | ❌ | 端口不同 |
| `http://a.com/x` | `http://a.com/y` | ✅ | 路径不影响 |
| `http://a.com` | `http://sub.a.com` | ❌ | 子域也算不同域 |

## 为什么需要同源策略

防止恶意网站窃取数据。没有同源策略的话：

```
1. 你登录了 bank.com，浏览器存了 cookie
2. 你访问 evil.com
3. evil.com 的 JS 发请求到 bank.com/api/balance
4. 浏览器自动带上 bank.com 的 cookie
5. evil.com 拿到你的余额 → 资金被盗
```

同源策略阻止第 4 步：evil.com 的脚本**能发出请求**，但**读不到响应**。

## 哪些操作受限制

```
┌─────────────────────────────────────────────┐
│            跨域时的实际情况                    │
├─────────────────────────────────────────────┤
│ ✅ 可以发出请求（请求确实到达了服务器）          │
│ ❌ 读不到响应（浏览器拦截了 response）          │
├─────────────────────────────────────────────┤
│ 不受同源策略限制的：                           │
│  • <script src="...">  → JSONP 的原理        │
│  • <link href="...">   → 加载 CSS            │
│  • <img src="...">     → 加载图片             │
│  • <video>/<audio>     → 加载媒体             │
│  • <form action="..."> → 表单提交             │
└─────────────────────────────────────────────┘
```

## 跨域解决方案

### 1. CORS（推荐）

**CORS（Cross-Origin Resource Sharing）** 是 W3C 标准，服务端设置响应头，浏览器自动处理。

```
客户端发请求
    │
    ▼
浏览器检查是否跨域
    │
    ├── 简单请求 → 直接发送，服务器返回 Access-Control-Allow-Origin
    │
    └── 非简单请求 → 先发 OPTIONS 预检请求
                         │
                         ├── 预检通过 → 发送真实请求
                         └── 预检失败 → 报错
```

#### 简单请求

满足以下条件：
- 方法：GET、HEAD、POST
- Content-Type：`text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`
- 无自定义请求头

```javascript
// 客户端正常发请求
fetch('https://api.other.com/data')
  .then(res => res.json())

// 服务端设置响应头
Access-Control-Allow-Origin: https://my-site.com  // 或 *
Access-Control-Allow-Credentials: true             // 允许携带 cookie
```

#### 预检请求（Preflight）

不满足简单请求条件时，浏览器自动先发 OPTIONS 请求：

```
浏览器 → OPTIONS /api/data → 服务器
         Origin: https://my-site.com
         Access-Control-Request-Method: PUT
         Access-Control-Request-Headers: Content-Type, Authorization

浏览器 ← 204 No Content ← 服务器
         Access-Control-Allow-Origin: https://my-site.com
         Access-Control-Allow-Methods: GET, POST, PUT, DELETE
         Access-Control-Allow-Headers: Content-Type, Authorization
         Access-Control-Max-Age: 86400              // 缓存 24 小时

浏览器 → PUT /api/data → 服务器    // 预检通过，发真实请求
```

#### CORS 响应头一览

| 响应头 | 作用 |
|---|---|
| `Access-Control-Allow-Origin` | 允许的源（`*` 或具体域名） |
| `Access-Control-Allow-Methods` | 允许的 HTTP 方法 |
| `Access-Control-Allow-Headers` | 允许的自定义请求头 |
| `Access-Control-Allow-Credentials` | 是否允许携带 cookie（`true`/`false`） |
| `Access-Control-Max-Age` | 预检结果缓存时间（秒） |
| `Access-Control-Expose-Headers` | 允许前端读取的响应头 |

#### 带 Cookie 的跨域

```javascript
// 客户端：必须设置 credentials
fetch('https://api.other.com/data', {
  credentials: 'include'  // 携带 cookie
})

// 服务端：
// Access-Control-Allow-Origin 不能用 *，必须是具体域名
// 必须设置 Access-Control-Allow-Credentials: true
Access-Control-Allow-Origin: https://my-site.com
Access-Control-Allow-Credentials: true
```

### 2. JSONP

利用 `<script>` 标签不受同源策略限制的特性。

```javascript
// 1. 定义回调函数
function handleData(data) {
  console.log(data)  // { name: '张三', age: 25 }
}

// 2. 动态创建 script 标签
const script = document.createElement('script')
script.src = 'https://api.other.com/user?callback=handleData'
document.body.appendChild(script)

// 3. 服务器返回：
// handleData({ "name": "张三", "age": 25 })

// 4. script 加载完后移除
script.onload = () => script.remove()
```

**JSONP 的缺点**：
| 缺点 | 说明 |
|---|---|
| 只支持 GET | 本质是 script 标签加载 |
| 安全风险 | 服务器返回的 JS 会被直接执行 |
| 错误处理弱 | 没有标准的错误回调机制 |
| 已过时 | CORS 是现代标准方案 |

### 3. 代理服务器

同源策略是**浏览器**的限制，服务器之间没有跨域问题。

```
浏览器 ──同源──→ 自己的服务器 ──无限制──→ 目标 API
         /api/proxy        http://other.com/data
```

**Webpack Dev Server 代理**：
```javascript
// webpack.config.js / vite.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.other.com',
        changeOrigin: true,    // 修改请求头中的 Origin
        pathRewrite: {
          '^/api': ''          // 去掉 /api 前缀
        }
      }
    }
  }
}

// 客户端请求 /api/users → 代理转发到 https://api.other.com/users
```

**Nginx 反向代理**：
```nginx
server {
    listen 80;
    server_name my-site.com;

    location /api/ {
        proxy_pass https://api.other.com/;
        proxy_set_header Host api.other.com;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 4. postMessage

跨窗口 / iframe 通信。

```javascript
// 父窗口（http://a.com）
const iframe = document.getElementById('myFrame')
iframe.contentWindow.postMessage('Hello from parent', 'http://b.com')

// 子窗口（http://b.com）
window.addEventListener('message', (e) => {
  if (e.origin === 'http://a.com') {  // 校验来源！
    console.log(e.data)  // 'Hello from parent'
    e.source.postMessage('Hello from child', e.origin)
  }
})
```

### 5. WebSocket

WebSocket 协议不受同源策略限制，但需要服务器支持。

```javascript
const ws = new WebSocket('wss://api.other.com/socket')
ws.onmessage = (e) => console.log(e.data)
```

## 各方案对比

| 方案 | 原理 | 方法限制 | 推荐度 |
|---|---|---|---|
| **CORS** | 服务端设置响应头 | 全部 | ⭐⭐⭐⭐⭐ |
| **代理** | 服务器中转，绕过浏览器限制 | 全部 | ⭐⭐⭐⭐⭐ |
| **JSONP** | script 标签不受同源限制 | 仅 GET | ⭐⭐ |
| **postMessage** | 跨窗口通信 API | - | ⭐⭐⭐ |
| **WebSocket** | 不受同源策略限制 | - | ⭐⭐⭐ |

## 常见面试问题

### 1. 什么是跨域？为什么会产生跨域？

同源策略限制了不同源之间的资源访问。协议、域名、端口任一不同即为跨域。目的是防止恶意网站窃取用户数据。

### 2. 跨域时请求能发出去吗？

**能**。请求确实到达了服务器并执行，服务器也返回了响应。是浏览器在**读取响应**时拦截了数据，并在控制台报错。

### 3. CORS 简单请求和预检请求的区别？

简单请求直接发送，浏览器在响应中检查 `Access-Control-Allow-Origin`。非简单请求先发 OPTIONS 预检请求，确认服务器允许后才发真实请求。预检请求会增加一次 RTT。

### 4. 为什么 `credentials: include` 时不能用 `Access-Control-Allow-Origin: *`？

安全原因。如果允许任意源携带 cookie，任何网站都能冒充用户身份访问 API。所以必须明确指定允许的源。

### 5. JSONP 和 CORS 的区别？

JSONP 只支持 GET，有安全风险，是 hack 方案。CORS 是 W3C 标准，支持所有 HTTP 方法，安全性更好，是现代推荐方案。

### 6. 预检请求（OPTIONS）会影响性能吗？

会多一次网络往返（RTT）。通过设置 `Access-Control-Max-Age` 缓存预检结果可以减少次数。在高频请求场景下，建议将 API 设计为简单请求以避免预检。

### 7. 跨域解决方案怎么选？

- **前后端分离开发**：用 CORS 或代理
- **生产环境**：Nginx 反向代理（同域部署最简单）
- **嵌入第三方页面通信**：postMessage
- **实时通信**：WebSocket
