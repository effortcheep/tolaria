# 浏览器渲染原理

## 一句话回答

浏览器从接收 HTML 到像素上屏，经过 **解析 → 构建 → 布局 → 绘制 → 合成** 五个阶段，每个阶段的输出是下一个阶段的输入，形成一条**渲染流水线（Rendering Pipeline）**。

---

## 问题一：从输入 URL 到页面渲染完成，中间经历了哪些过程？

整个过程分为**网络阶段**和**渲染阶段**两大块。

### 网络阶段（获取资源）

```
用户输入 URL
    ↓
浏览器进程判断是搜索关键词还是 URL
    ↓
URL 补全（自动加 https:// 等）
    ↓
浏览器进程 → 网络进程
    ↓
DNS 解析：域名 → IP 地址
  ├── 浏览器 DNS 缓存 → 命中直接返回
  ├── 本地 DNS 缓存 → 命中直接返回
  └── 递归查询：根 DNS → 顶级域 DNS → 权威 DNS
    ↓
TCP 三次握手（建立连接）
  ├── 如果是 HTTPS → TLS 握手（协商密钥）
  └── HTTP/2 → 多路复用，一个连接传多个请求
    ↓
发送 HTTP 请求（GET /index.html）
    ↓
服务器返回响应（状态码 + 响应头 + 响应体）
    ↓
网络进程判断 Content-Type
  ├── text/html → 交给渲染进程处理
  ├── application/json → 交给 JS 处理
  └── application/octet-stream → 触发下载
```

**关键节点说明：**

| 节点 | 说明 |
|------|------|
| DNS 解析 | 域名 → IP，递归查询最多 8 个来回 |
| TCP 三次握手 | SYN → SYN+ACK → ACK，确保双方都能收发 |
| TLS 握手 | 非对称加密交换密钥，后续用对称加密通信 |

### 渲染阶段（绘制页面）

```
HTML 字节流
    ↓ 解码
HTML 字符串
    ↓ Token 化（状态机逐字节扫描）
HTML Token（StartTag、EndTag、Text）
    ↓ 栈结构构建节点关系
DOM 树
    ↓ + CSSOM 树合并
渲染树（Render Tree，只包含可见元素）
    ↓ 布局（Layout/Reflow）
布局树（Layout Tree，带位置和尺寸信息）
    ↓ 分层（Layer）
图层树（Layer Tree）
    ↓ 绘制（Paint）
绘制记录（Paint Records，绘制指令序列）
    ↓ 光栅化（Raster）
位图（Bitmap）
    ↓ 合成（Composite）
屏幕像素
```

#### 第一步：解析 HTML，构建 DOM 树

**网络进程 → 渲染进程：**

- 浏览器进程接收到 URL，交给**网络进程**发起请求
- 网络进程收到响应头，根据 `Content-Type` 判断是 HTML
- 为该标签页创建**渲染进程**，通过**共享内存**将 HTML 字节流传给渲染进程

**Token 化（Tokenization）：**

HTML 不是上下文无关语法，不能用常规解析器。浏览器用**状态机**逐字节扫描：

```
<div class="box">hello</div>
→ StartTag(div, class="box") → Text("hello") → EndTag(div)
```

- **起始标签 Token**：记录标签名、属性
- **结束标签 Token**：记录标签名
- **文本 Token**：记录文本内容

**构建 DOM：**

每个 Token 通过**栈结构**构建节点关系：

```
栈操作过程：
1. StartTag(div)   → 入栈 → 栈: [html, body, div]
2. StartTag(span)  → 入栈 → 栈: [html, body, div, span]
3. Text("hello")   → span 的子节点
4. EndTag(span)    → 出栈 → 栈: [html, body, div]
5. EndTag(div)     → 出栈 → 栈: [html, body]
```

> 遇到 `<script>` 会阻塞 DOM 解析（除非 async/defer）

#### 第二步：解析 CSS，构建 CSSOM 树

- CSS 加载**不会阻塞 DOM 解析**，但会**阻塞渲染**
- CSS 引擎从**右向左**匹配选择器（右边更具体，能更快缩小范围）

```
用户代理样式（浏览器默认）
    ↓ 覆盖
作者样式表（开发者写的 CSS）
    ↓ 覆盖
用户样式表（用户自定义）
    ↓
最终计算样式
```

#### 第三步：生成渲染树（Render Tree）

DOM + CSSOM 合并，只包含**可见元素**：

```
DOM 树          CSSOM 树
  ↓                ↓
  └── 合并 ──────┘
        ↓
   渲染树（只包含可见元素）
```

**关键规则：**
- `display: none` → **不生成**渲染树节点
- `visibility: hidden` → **生成**渲染树节点（占位，只是不画）
- 伪元素（`::before`、`::after`）→ **生成**渲染树节点

#### 第四步：布局（Layout / Reflow）

遍历渲染树，为每个节点计算**位置**（x, y）和**尺寸**（width, height），生成**布局树**。

#### 第五步：分层（Layer）

某些 CSS 属性会创建独立的**合成层**，某一层变化时只需重绘该层：

| 类型 | 触发条件 |
|------|----------|
| 滚动内容层 | `overflow: scroll` 的容器 |
| 定位层 | `position: fixed` / `sticky` |
| CSS 动画层 | `will-change` / `transform: translateZ(0)` |
| 透明度层 | `opacity < 1` + 合成条件 |
| Canvas/WebGL | 2D/3D 绘图上下文 |
| 视频层 | `<video>` 标签 |

#### 第六步：绘制（Paint）

渲染引擎对图层生成**绘制记录（Paint Records）**——绘制指令序列，不是真正的像素：

```
1. 绘制背景色
2. 绘制背景图片
3. 绘制边框
4. 绘制文字
5. 绘制阴影
```

#### 第七步：光栅化（Raster）

将绘制记录转换为**位图（Bitmap）**：

- 浏览器将图层分成**图块（Tile）**，通常 256×256 或 512×512 像素
- **光栅线程**将图块转为位图，优先处理视口附近的图块
- GPU 光栅化比 CPU 光栅化更快

#### 第八步：合成（Composite）与显示

**合成线程**负责：

1. 收集所有图层的**图块绘制指令（Draw Quad）**
2. 生成**合成帧（Compositor Frame）**
3. 通过 IPC 提交给**浏览器进程**
4. 浏览器进程将合成帧发送给 **GPU**
5. GPU 渲染到屏幕上

## 问题二：什么是重排（Reflow）和重绘（Repaint）？分别什么时候触发？

### 重排（Reflow / Layout）

**定义：** 元素的**几何属性**（位置、尺寸）发生变化，浏览器需要重新计算布局树中受影响节点的几何信息。

**本质：** 重新计算元素在页面中的位置和大小。

**触发条件：**

| 操作 | 为什么触发重排 |
|------|----------------|
| 修改 width/height/margin/padding/border | 尺寸变化，影响自身和周围元素 |
| 增删 DOM 元素 | 树结构变了，需要重新计算 |
| 读取 offsetWidth/offsetTop/clientWidth/scrollTop | **强制同步布局**，浏览器必须立即计算最新值 |
| 修改字体大小 | 影响文字排版，可能改变元素尺寸 |
| 窗口 resize | 视口变化，所有元素可能都要重新计算 |
| 修改 display | 从 none 变为可见，需要重新布局 |
| 激活 CSS 伪类（:hover） | 可能导致样式变化 |
| 修改类名导致样式变化 | 间接触发几何属性变化 |

**重排的影响范围：**

```
某个节点变化
    ├── 自身重新计算
    ├── 所有子节点重新计算（子节点位置依赖父节点）
    └── 可能影响兄弟节点（如浮动元素、flex 布局）
```

**重排的成本很高**——一个节点变化可能引发整棵子树的重新计算。

### 重绘（Repaint）

**定义：** 元素的**外观属性**（颜色、背景、阴影等）发生变化，但不影响布局，浏览器只需要重新绘制受影响的像素。

**触发条件：**

| 操作 | 为什么只触发重绘 |
|------|------------------|
| 修改 color | 外观变化，不影响布局 |
| 修改 background-color / background-image | 外观变化 |
| 修改 box-shadow / text-shadow | 外观变化 |
| 修改 outline | 外观变化 |
| 修改 visibility | 隐藏但仍然占位，不影响布局 |
| 修改 border-radius | 外观变化（不改变尺寸时） |

### 两者的关系

```
重排（Reflow）一定触发重绘（Repaint）
重绘（Repaint）不一定触发重排（Reflow）
```

```
几何属性变化（位置、尺寸）
    ↓
重排（重新计算布局）
    ↓
重绘（重新绘制像素）

外观属性变化（颜色、背景）
    ↓
重绘（重新绘制像素）
    ↓ 不需要重排
```

**类比：**
- **重排** = 把房间里的家具全部重新摆放（位置变了）
- **重绘** = 把墙重新刷一遍颜色（位置没变，只是外观变了）

### 更精细的渲染阶段划分

现代浏览器把渲染过程分得更细：

```
JavaScript（触发样式变化）
    ↓
Style（计算样式）—— 样式计算
    ↓
Layout（布局）—— 重排
    ↓
Paint（绘制）—— 重绘
    ↓
Composite（合成）—— 合成
```

| 阶段 | 触发的 CSS 属性 | 性能成本 |
|------|-----------------|----------|
| Layout | width, height, margin, padding, position, display, border, font-size | **最高** |
| Paint | color, background, box-shadow, outline, visibility | **中等** |
| Composite | transform, opacity | **最低** |

## 问题三：如何减少重排和重绘？

### 策略一：用 transform 代替 top/left（优先级最高）

`top/left` 触发重排，`transform` 只触发合成，由 GPU 直接处理：

```javascript
// ❌ 差：触发 Layout → Paint → Composite
box.style.left = '100px'
box.style.top = '50px'

// ✅ 好：只触发 Composite
box.style.transform = 'translate(100px, 50px)'
```

同理，用 `opacity` 代替 `visibility`：

```css
/* opacity 只触发合成，visibility 触发重绘 */
.box { opacity: 0; }  /* 更优 */
.box { visibility: hidden; }
```

### 策略二：批量修改样式

```javascript
// ❌ 差：3 次重排
box.style.width = '100px'
box.style.height = '100px'
box.style.margin = '10px'

// ✅ 好：1 次重排
box.style.cssText = 'width: 100px; height: 100px; margin: 10px;'

// ✅ 更好：通过 class 批量切换
box.classList.add('new-style')
```

浏览器会把同步代码中连续的样式修改合并为一次重排。如果中间插入了读取操作（如 `offsetWidth`），会强制立即重排。

### 策略三：离线 DOM 操作

```javascript
// ❌ 差：每次 appendChild 都触发重排
for (let i = 0; i < 1000; i++) {
  container.appendChild(createItem(i))
}

// ✅ 好：使用 DocumentFragment
const fragment = document.createDocumentFragment()
for (let i = 0; i < 1000; i++) {
  fragment.appendChild(createItem(i))
}
container.appendChild(fragment)  // 只触发 1 次重排

// ✅ 好：先隐藏，操作完再显示
container.style.display = 'none'
for (let i = 0; i < 1000; i++) {
  container.appendChild(createItem(i))
}
container.style.display = ''  // 只触发 1 次重排
```

`DocumentFragment` 是内存中的轻量级 DOM 容器，不参与页面布局，只有插入页面时才触发一次重排。

### 策略四：避免强制同步布局

```javascript
// ❌ 差：读写交替，强制同步布局
box.style.width = '100px'
console.log(box.offsetWidth)  // 强制回流获取最新值
box.style.height = '100px'   // 再次触发回流

// ✅ 好：先读完再写
const width = box.offsetWidth  // 读
box.style.width = '100px'      // 写
box.style.height = '100px'     // 写
```

读取 `offsetWidth` 时，浏览器必须保证返回最新值，会暂停 JS 执行立即触发重排，这叫**强制同步布局（Forced Synchronous Layout）**。

### 策略五：使用 requestAnimationFrame

```javascript
// ❌ 差：在渲染中途修改样式
setTimeout(() => {
  box.style.transform = `translateX(${x}px)`
}, 16)

// ✅ 好：在下一帧渲染前修改样式
requestAnimationFrame(() => {
  box.style.transform = `translateX(${x}px)`
})
```

`requestAnimationFrame` 的回调在浏览器**重绘之前**执行，确保样式修改和渲染节奏同步。

### 策略六：使用 will-change 提前通知浏览器

```css
.box {
  will-change: transform;  /* 浏览器提前创建独立图层 */
}
```

浏览器会提前为该元素创建独立的合成层，后续 `transform` 变化只影响该层。但不要滥用——每个图层都消耗 GPU 内存，用完后应移除。

### 策略七：使用 contain 限制影响范围

```css
.card {
  contain: layout;  /* 该元素的布局变化不影响外部 */
}
```

`contain` 告诉浏览器该元素是**自包含的**，内部变化不会影响外部元素的布局，浏览器可以跳过对其他节点的重排计算。

### 优化优先级总结

| 优先级 | 策略 | 效果 |
|--------|------|------|
| 1 | 用 `transform` 代替 `top/left`，`opacity` 代替 `visibility` | 避免重排，GPU 加速 |
| 2 | 批量修改样式（class 切换） | 减少重排次数 |
| 3 | 离线 DOM 操作（DocumentFragment） | 减少重排次数 |
| 4 | 读写分离，避免强制同步布局 | 避免额外重排 |
| 5 | `requestAnimationFrame` | 同步渲染节奏 |
| 6 | `will-change` / `contain` | 提前优化，限制影响范围 |

## 补充：其他常见问题

### CSS 放在 head 里，JS 放在 body 底部？

- CSS 放 head：尽早下载，避免 FOUC（无样式闪烁）
- JS 放底部：JS 会**阻塞 DOM 解析**（因为 JS 可能操作 DOM），放在底部减少阻塞时间
- `<script async>`：异步下载，下载完立即执行（执行时仍阻塞 DOM）
- `<script defer>`：异步下载，等 DOM 解析完再执行（按顺序）

### display: none 和 visibility: hidden 的区别？

| 特性 | display: none | visibility: hidden |
|------|---------------|-------------------|
| 渲染树 | 不生成节点 | 生成节点 |
| 占位 | 不占位 | 占位 |
| 回流 | 触发 | 不触发 |
| 重绘 | 触发 | 触发 |
| 子元素 | 全部隐藏 | 可单独设置 visible |

### 什么是 FOUC？如何避免？

**FOUC（Flash of Unstyled Content）**：页面先显示无样式内容，CSS 加载后突然变样式。

**避免方法：**
- CSS 放在 `<head>` 中
- 使用 `<link>` 而非 `@import`（`@import` 是渲染阻塞的）
- 关键 CSS 内联到 HTML 中
