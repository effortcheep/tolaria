# 浏览器存储

## 核心概念梳理

浏览器存储是前端在客户端持久化/临时保存数据的机制，主要有三种 API：**Cookie**、**LocalStorage**、**SessionStorage**。此外还有 **IndexedDB**（大量结构化数据）和 **Cache Storage**（网络请求缓存），但面试重点是前三者。

### 存储位置与生命周期

```
┌─────────────────────────────────────────────────┐
│                 浏览器存储体系                    │
├──────────────┬──────────────┬───────────────────┤
│    Cookie     │ LocalStorage│  SessionStorage   │
├──────────────┼──────────────┼───────────────────┤
│ 服务端+客户端 │   纯客户端   │    纯客户端        │
│ 每次请求携带  │  永久存储    │  会话级（标签页）   │
│ 4KB 限制     │  5~10MB     │  5~10MB           │
│ 键值对字符串  │  键值对字符串 │  键值对字符串      │
└──────────────┴──────────────┴───────────────────┘
```

### 共同点

- 都是**键值对**存储，值只能是**字符串**（对象需 `JSON.stringify`）
- 都遵循**同源策略**（协议+域名+端口相同）
- 都在浏览器端，不跨浏览器共享

## 1. Cookie、LocalStorage、SessionStorage 有什么区别？

| 维度 | Cookie | LocalStorage | SessionStorage |
|------|--------|-------------|---------------|
| **设计目的** | 服务端状态管理，HTTP 无状态的补充 | 纯客户端持久存储 | 纯客户端会话级存储 |
| **生命周期** | 可设 Expires/Max-Age；不设则浏览器关闭失效 | **永久存储**，除非手动删除 | **标签页关闭即失效**（刷新不丢） |
| **容量** | **4KB** | **5~10MB** | **5~10MB** |
| **与请求的关系** | **每次 HTTP 请求自动携带**（Cookie 头） | 不自动携带 | 不自动携带 |
| **可访问性** | 同源 + 同路径（Path 可控） | 同源 | 同源 + **同一标签页** |
| **跨标签页** | ✅ 共享 | ✅ 同源标签页共享 | ❌ 每个标签页独立 |
| **API** | `document.cookie`（笨拙，需手动解析） | `localStorage.getItem/setItem` | `sessionStorage.getItem/setItem` |
| **安全性** | 可设 HttpOnly 阻止 JS 访问 | JS 可直接读写，XSS 可窃取 | JS 可直接读写，XSS 可窃取 |
| **服务端参与** | ✅ 服务端可通过 Set-Cookie 写入 | ❌ 服务端无感知 | ❌ 服务端无感知 |

### 关键差异总结

1. **Cookie vs Web Storage**：Cookie 是为服务端通信设计的，每次请求自动携带，因此容量小且浪费带宽；Web Storage 纯客户端使用，容量大且不影响网络请求
2. **LocalStorage vs SessionStorage**：唯一核心区别是**生命周期**——LocalStorage 永久，SessionStorage 仅限当前标签页会话

## 2. Cookie 的属性有哪些？

```
Set-Cookie: name=value; Domain=.example.com; Path=/; 
            Expires=Thu, 01 Jan 2026 00:00:00 GMT; 
            HttpOnly; Secure; SameSite=Lax
```

| 属性 | 作用 | 说明 |
|------|------|------|
| **Name=Value** | 键值对 | 值含特殊字符需 URL 编码 |
| **Domain** | Cookie 所属域名 | 设为 `.example.com` 则所有子域共享；不设默认当前域（不含子域） |
| **Path** | Cookie 可用路径 | `/api` 表示只有 `/api` 及其子路径下的请求才携带 |
| **Expires** | 绝对过期时间 | `Thu, 01 Jan 2026 00:00:00 GMT`；不设则为**会话 Cookie**（浏览器关闭删除） |
| **Max-Age** | 相对过期时间（秒） | `Max-Age=3600` = 1 小时后过期；**优先级高于 Expires** |
| **HttpOnly** | 禁止 JS 访问 | `document.cookie` 读不到，**防 XSS 窃取 Cookie**；仍随请求发送 |
| **Secure** | 仅 HTTPS 传输 | HTTP 页面不会携带此 Cookie |
| **SameSite** | 跨站请求限制 | **防 CSRF**，三个值见下文 |

### SameSite 详解（重点）

| 值 | 行为 | 适用场景 |
|----|------|---------|
| **Strict** | 跨站请求**完全不携带** Cookie | 安全性最高，但用户体验差（从外部链接点进来不带登录态） |
| **Lax**（默认） | 跨站的 GET 导航（链接、预加载）携带，POST/iframe/img 等**不携带** | 安全与体验的平衡，大多数场景推荐 |
| **None** | 跨站请求**始终携带** | 必须同时设 `Secure`，用于第三方 Cookie（如嵌入式 iframe） |

### Cookie 与安全相关的最佳实践

- 敏感 Cookie：`HttpOnly` + `Secure` + `SameSite=Lax`
- 防 XSS：`HttpOnly` 阻止 JS 读取
- 防 CSRF：`SameSite=Lax/Strict` 阻止跨站携带
- 防窃听：`Secure` 确保仅 HTTPS 传输

## 3. 什么场景用哪种存储？

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| **登录认证**（传统方案） | Cookie（HttpOnly） | 请求自动携带，服务端可验证，JS 无法窃取 |
| **JWT Token** | LocalStorage 或 Cookie | LocalStorage 容量大但有 XSS 风险；Cookie 更安全但有 CSRF 风险 |
| **用户偏好设置**（主题、语言） | LocalStorage | 永久保存，纯客户端使用，不需要发给服务端 |
| **表单草稿/临时数据** | SessionStorage | 标签页关闭自动清除，不污染其他会话 |
| **多标签页独立状态**（多页编辑器） | SessionStorage | 各标签页互不影响 |
| **缓存 API 响应** | LocalStorage / IndexedDB | 持久化缓存，减少网络请求 |
| **大量结构化数据** | IndexedDB | LocalStorage 只能存字符串，IndexedDB 支持索引和事务 |
| **服务端需要读取的状态** | Cookie | 唯一能随请求自动发送的方案 |

### 选择决策树

```
需要发给服务端？
├── 是 → Cookie（设 HttpOnly + Secure + SameSite）
└── 否 → 数据量大/结构化？
    ├── 是 → IndexedDB
    └── 否 → 需要持久保存？
        ├── 是 → LocalStorage
        └── 否 → SessionStorage
```

### 实际开发中的典型组合

- **认证**：Cookie（HttpOnly）存 Session ID，或 LocalStorage 存 JWT + 请求头 Authorization
- **缓存**：LocalStorage 存用户配置 + IndexedDB 存业务数据
- **多标签页**：SessionStorage 隔离各标签页的临时状态
