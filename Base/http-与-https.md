# HTTP 与 HTTPS

## 什么是 HTTP

**HTTP**（HyperText Transfer Protocol，超文本传输协议）是互联网上应用最广泛的**应用层协议**，用于客户端与服务器之间的**请求-响应**通信。

```
客户端 ──HTTP 请求──▶ 服务器
客户端 ◀──HTTP 响应── 服务器
```

## 什么是 HTTPS

**HTTPS**（HTTP Secure）= **HTTP + SSL/TLS**，在 HTTP 的基础上增加了**加密**和**身份验证**。

```
客户端 ──TLS 握手──▶ 服务器    （验证身份、协商密钥）
客户端 ──加密的 HTTP──▶ 服务器  （数据加密传输）
客户端 ◀─加密的 HTTP── 服务器
```

## 核心区别对比

| 特性 | HTTP | HTTPS |
|------|------|-------|
| **协议** | HTTP | HTTP + SSL/TLS |
| **默认端口** | 80 | 443 |
| **数据传输** | 明文（裸奔） | 加密 |
| **安全性** | ❌ 无加密 | ✅ 加密 + 身份验证 |
| **证书** | 不需要 | 需要 CA 证书 |
| **性能** | 快（无额外开销） | 略慢（TLS 握手 + 加解密） |
| **SEO** | 不加分 | 搜索引擎优先收录 |
| **URL 前缀** | `http://` | `https://` |

## HTTPS 解决的三大问题

### 1. 窃听（Eavesdropping）— 机密性

```
❌ HTTP：数据明文传输
   客户端 ──密码=123456──▶ 服务器
              ↑
           中间人直接看

✅ HTTPS：数据加密传输
   客户端 ──a]X#k$9f──▶ 服务器
              ↑
           中间人看不懂
```

### 2. 篡改（Tampering）— 完整性

```
❌ HTTP：数据可被修改
   客户端 ──"转100元"──▶ 服务器
              ↓
           中间人改成"转10000元"
              ↓
         服务器收到"转10000元"

✅ HTTPS：数据带摘要，篡改后校验失败
   客户端 ──"转100元"+MAC──▶ 服务器
              ↓
           中间人改了内容，但没有 MAC 密钥
              ↓
         服务器校验失败，拒绝请求
```

### 3. 冒充（Impersonation）— 身份认证

```
❌ HTTP：无法确认服务器身份
   客户端 ──▶ 黑客伪造的网站（长得一模一样）

✅ HTTPS：CA 证书验证身份
   客户端 ──▶ 检查证书 ──▶ 证书由可信 CA 签发 ──▶ 确认是真实服务器
```

## TLS 握手过程

HTTPS 在传输数据前需要进行一次 **TLS 握手**，协商加密参数：

```
客户端                                          服务器
  │                                               │
  │── ClientHello ──────────────────────────────▶ │  ① 发送支持的 TLS 版本、加密套件、随机数
  │                                               │
  │◀────────────────────────── ServerHello ────── │  ② 选择加密套件、发送随机数
  │◀────────────────────────── Certificate ────── │  ③ 发送服务器证书（公钥）
  │◀────────────────────────── ServerHelloDone ── │
  │                                               │
  │── 验证证书 ── 生成预主密钥 ── 用公钥加密 ──   │  ④ 客户端验证证书，生成预主密钥
  │── ClientKeyExchange ────────────────────────▶ │  ⑤ 发送加密的预主密钥
  │── ChangeCipherSpec ─────────────────────────▶ │  ⑥ 切换到加密通信
  │── Finished ────────────────────────────────▶  │  ⑦ 握手完成
  │                                               │
  │◀─────────────────────── ChangeCipherSpec ──── │
  │◀─────────────────────── Finished ──────────── │
  │                                               │
  │═══════════ 加密通信开始 ═════════════════════ │
```

**三个随机数**：
- 客户端随机数（明文传输）
- 服务端随机数（明文传输）
- 预主密钥（客户端生成，用服务器公钥加密传输）

三个随机数通过**伪随机函数**生成最终的**会话密钥**，用于对称加密。

## 加密方式

### 对称加密

```
加密：密文 = 加密算法(明文, 密钥)
解密：明文 = 解密算法(密文, 密钥)
```

- 特点：加密和解密用**同一个密钥**
- 优点：速度快
- 缺点：密钥传输不安全
- 常见算法：AES、ChaCha20

### 非对称加密

```
加密：密文 = 加密算法(明文, 公钥)
解密：明文 = 解密算法(密文, 私钥)
```

- 特点：公钥加密，只有私钥能解密（反过来也行）
- 优点：公钥可以公开，解决了密钥分发问题
- 缺点：速度慢（比对称加密慢 100~1000 倍）
- 常见算法：RSA、ECDHE

### HTTPS 的组合策略

```
TLS 握手阶段：非对称加密（安全地交换密钥）
数据传输阶段：对称加密（高效地加密数据）

取两者之长，避两者之短
```

## CA 证书

### 什么是 CA

**CA**（Certificate Authority）是受信任的**证书颁发机构**，负责验证网站身份并签发数字证书。

### 证书链

```
根 CA（Root CA）
  └── 中间 CA（Intermediate CA）
        └── 网站证书（Leaf Certificate）
```

浏览器内置了**根 CA 证书列表**，信任从根 CA 一层层传递下来。

### 证书验证过程

```
1. 客户端收到服务器证书
2. 检查证书是否过期
3. 检查证书域名是否匹配
4. 用中间 CA 的公钥验证网站证书的签名
5. 用根 CA 的公钥验证中间 CA 的签名
6. 根 CA 在本地信任列表中 → 证书可信
```

## 常见面试问题

### 1. HTTP 和 HTTPS 最大的区别是什么？

> HTTPS 在 HTTP 的基础上加入了 SSL/TLS 层，提供了**加密**（防窃听）、**完整性校验**（防篡改）、**身份认证**（防冒充）三大安全保障。

### 2. HTTPS 的加密过程是怎样的？

> 采用**混合加密**：
> 1. TLS 握手阶段用**非对称加密**（RSA/ECDHE）安全交换密钥
> 2. 后续通信用**对称加密**（AES）高效加密数据
> 3. 通过**数字证书**验证服务器身份，防止中间人攻击

### 3. 为什么不全部用非对称加密？

> 非对称加密计算量大，速度比对称加密慢 100~1000 倍。大数据量传输时性能不可接受。所以只在握手阶段用非对称加密交换密钥，之后用对称加密。

### 4. HTTPS 能被抓包吗？

> **能**。抓包工具（如 Charles、Fiddler）的工作原理是**中间人代理**：
> - 代理自己生成证书，冒充服务器
> - 客户端需要**手动安装并信任**代理的根证书
> - 如果不装证书，抓到的是密文，无法解密
>
> 所以 HTTPS 不是"不能被抓"，而是"不能被偷偷抓"。

### 5. 什么是中间人攻击？

> 攻击者拦截通信，冒充服务器与客户端通信，同时冒充客户端与服务器通信：
>
> ```
> 客户端 ◀──▶ 攻击者 ◀──▶ 服务器
>          都以为在和对方直接通信
> ```
>
> HTTPS 通过**证书验证**防止中间人攻击——攻击者无法获得 CA 签发的合法证书。

### 6. 为什么不默认用 HTTPS？

> 早期的顾虑（现在基本已消除）：
> - **成本**：证书需要钱（现在 Let's Encrypt 免费）
> - **性能**：TLS 握手有开销（现在 TLS 1.3 只需 1-RTT，硬件加速普及）
> - **兼容性**：老旧设备不支持（现在几乎不存在）
>
> 现在主流网站已全面转向 HTTPS，Chrome 会把 HTTP 标记为"不安全"。

### 7. TLS 1.2 和 1.3 有什么区别？

> | 特性 | TLS 1.2 | TLS 1.3 |
> |------|---------|---------|
> | 握手 RTT | 2-RTT | 1-RTT（0-RTT 恢复） |
> | 密钥交换 | RSA / DHE / ECDHE | 只保留 ECDHE |
> | 对称加密 | AES-CBC / AES-GCM | 只保留 AEAD（AES-GCM / ChaCha20） |
> | 安全性 | 支持弱算法 | 砍掉所有弱算法 |
> | 握手明文 | 部分明文 | 全部加密 |
